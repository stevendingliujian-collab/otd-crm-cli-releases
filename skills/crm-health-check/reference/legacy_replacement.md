# Playbook：legacy_replacement
# 适用场景：替换旧系统 / 替换竞品项目

## 行业特征

- 客户已有系统在用，替换决策比新建更难（涉及历史数据、用户习惯、替换风险）
- 核心问题：为什么非换不可？换的阻力来自哪里？
- 数据迁移和并行运行是必须面对的技术问题
- 替换旧系统往往意味着旧供应商也在参与竞争（关系维护）

## 推荐阶段停留周期

| 阶段 | 建议天数 | 超1.2倍 | 超1.5倍 | 超2倍 |
|------|----------|---------|---------|-------|
| 初步接触 | 7–14 天 | 轻度预警 | 中度预警 | 重度预警 |
| 需求确认 | 10–21 天 | 轻度预警 | 中度预警 | 重度预警 |
| 调研/诊断 | 14–30 天 | 轻度预警 | 中度预警 | 重度预警 |
| 方案设计 | 10–25 天 | 轻度预警 | 中度预警 | 重度预警 |
| 报价/商务 | 14–35 天 | 轻度预警 | 中度预警 | 重度预警 |
| 谈判/采购 | 20–50 天 | 轻度预警 | 中度预警 | 重度预警 |
| 待签约 | 10–21 天 | 轻度预警 | 中度预警 | 重度预警 |

## 各阶段关键角色要求

| 阶段 | 至少应识别的角色 |
|------|-----------------|
| 初步接触 | 提出替换需求的角色（通常是使用部门或IT） |
| 需求确认 | IT 负责人、使用部门负责人 |
| 调研/诊断 | IT、使用部门、财务（旧系统维护成本） |
| 方案设计 | 决策者、IT、使用部门 |
| 报价/商务 | 决策者、IT（技术可行性确认） |
| 谈判/采购 | 拍板人、法务/采购 |

## 关键判断点（必须在调研阶段搞清楚）

1. **现有系统是谁家的**（旧供应商的维护合同和关系现状）
2. **客户为什么想换**（功能不足？服务差？价格贵？技术老旧？）
3. **不满点**：功能、服务、交付、价格哪个是主要矛盾
4. **替换阻力来自哪里**（IT 不想换？使用部门担心迁移影响生产？老板关系户？）
5. **数据迁移范围**：历史数据是否需要迁移，量有多大
6. **切换方式**：并行运行多久？如何平滑过渡？

## 最佳实践

### 加分行为
- 明确"为什么非换不可"——让客户自己说出来
- 输出低风险替换方案（并行运行、分批迁移）
- 尽早识别历史数据迁移、接口切换、并行运行安排
- 提供同类替换项目的成功案例
- 不只讲"我们更强"，重点讲"如何低风险替换"

### 减分行为
- 只强调我们产品比旧系统好，不解决替换风险
- 数据迁移方案不清晰就报价
- 未识别旧供应商的竞争动作（旧供应商可能降价或改善服务）
- 未确认内部替换阻力的来源和化解方式

## 当前阶段卡住时的推荐动作

### 卡在需求确认
1. 直接问"现在系统最让你们头疼的是什么"，让客户列举
2. 确认替换是使用部门主导还是IT/管理层主导
3. 了解旧系统维护合同状态（是否快到期？续费贵？）
4. 识别最大的替换阻力（技术/人/关系），评估可突破性

### 卡在方案设计
1. 方案中必须包含"切换方案"章节（并行运行计划）
2. 明确数据迁移范围和方式
3. 对比旧系统不足 + 新方案改善，一一对应
4. 给出分阶段替换路径（降低一次性替换风险）
