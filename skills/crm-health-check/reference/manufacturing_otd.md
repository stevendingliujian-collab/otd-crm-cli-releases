# Playbook：manufacturing_otd
# 适用场景：OTD / 供产销协同 / 计划交付 / 订单到交付

## 行业特征

- 涉及销售、计划、生产、采购、仓储多部门协同
- 核心价值：交期承诺、计划变更响应、齐套与缺料管理
- 痛点通常在"信息割裂"而非某个单一部门
- 项目起点可以是订单协同，也可以是交付透明化，需明确
- 组织协调难度大，决策层需早期介入

## 推荐阶段停留周期

| 阶段 | 建议天数 | 超1.2倍 | 超1.5倍 | 超2倍 |
|------|----------|---------|---------|-------|
| 初步接触 | 7–14 天 | 轻度预警 | 中度预警 | 重度预警 |
| 需求确认 | 10–21 天 | 轻度预警 | 中度预警 | 重度预警 |
| 调研/诊断 | 14–30 天 | 轻度预警 | 中度预警 | 重度预警 |
| 方案设计 | 10–25 天 | 轻度预警 | 中度预警 | 重度预警 |
| 报价/商务 | 14–35 天 | 轻度预警 | 中度预警 | 重度预警 |
| 谈判/采购 | 15–45 天 | 轻度预警 | 中度预警 | 重度预警 |
| 待签约 | 7–21 天 | 轻度预警 | 中度预警 | 重度预警 |

## 各阶段关键角色要求

| 阶段 | 至少应识别的角色 |
|------|-----------------|
| 初步接触 | 窗口人（通常是运营/计划/销售负责人） |
| 需求确认 | 计划/运营负责人、销售/订单负责人 |
| 调研/诊断 | 计划、销售、采购、生产、IT 各部门代表 |
| 方案设计 | 运营总监/副总、各部门负责人、IT |
| 报价/商务 | 决策者（总经理/运营副总）、IT |
| 谈判/采购 | 拍板人、采购/法务 |

## 关键推进证据

- 已识别订单、计划、采购、生产、发货链条中的关键断点
- 已明确项目起点（订单协同？交期承诺？计划拉通？）
- 已识别核心管理痛点（不是泛"要数字化"）
- 已明确是否从试点事业部/工厂切入
- 已识别现有系统（ERP、MES、WMS）的边界和数据断点

## 常见风险

- 需求太泛（"我们要打通全流程"），无法落地
- 涉及部门太多，内部对齐困难，项目推进缓慢
- 未识别现有系统边界，集成方案不清晰
- 决策层没有介入，项目容易被搁置

## 最佳实践

### 加分行为
- 已完成跨部门调研（不只听一个部门的声音）
- 已绘制订单到交付的流程断点图
- 已识别"最痛的那一个断点"作为一期切入
- 已确认是否从单工厂/单事业部试点
- 已让管理层参与问题确认

### 减分行为
- 只听一个部门的需求就做方案
- 方案覆盖全流程，未做优先级拆分
- 未识别现有 ERP/MES 边界就设计数据流
- 没有试点策略，一次性要求全公司上线

## 当前阶段卡住时的推荐动作

### 卡在需求确认
1. 不要只听一个部门，必须跨部门收集信息
2. 画出从"客户下单"到"货物出厂"的全流程，找断点
3. 识别最核心的3个管理痛点（不超过3个）
4. 确认是否可以先从某一个事业部或工厂试点

### 卡在方案设计
1. 方案必须有"一期切入点"，不能是全流程大方案
2. 明确与现有 ERP/MES 系统的边界和接口
3. 拉运营/计划负责人确认方案是否解决了核心痛点
4. 让管理层看方案，重点讲"改善后的经营成效"
